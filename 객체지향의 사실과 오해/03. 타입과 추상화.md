📌 3장. 타입과 추상화

# 1. 추상화란 무엇인가?
	•	현실은 복잡성의 괴물 → 인간은 모든 정보를 그대로 처리할 수 없음.
	•	그래서 불필요한 세부 사항을 감추고, 중요한 것만 남기는 것이 추상화.
	•	추상화의 목적 = 복잡성을 이해 가능한 수준으로 단순화하는 것.
	•	추상화의 두 가지 방식:
	1.	일반화: 공통점만 취하고 차이점은 제거
	2.	세부 제거: 목적에 필요 없는 정보는 감추기

👉 객체지향은 바로 이 추상화의 도구로서 객체를 사용한다.

---

# 2. 객체지향과 추상화

● 그룹으로 나누어 단순화하기
	•	서로 명확히 구별되는 사물 → 객체
	•	“정원사, 병사, 왕, 여왕”을 모두 트럼프로 묶는 것 = 추상화(일반화)

● 개념(concept)
	•	많은 객체를 묶어 의미를 부여하는 일종의 ‘그릇’
	•	구성 요소:
	•	심볼(Symbol): 개념의 이름
	•	내연(Intension): 개념에 속할 조건
	•	외연(Extension): 그 개념에 해당하는 객체의 집합

● 분류(Classification)
	•	어떤 객체가 어떤 개념에 속하는지 결정하는 과정
	•	좋은 분류 체계는 유지보수 용이 + 머릿속 정신 모델 제공

👉 개념과 분류는 모두 추상화를 위한 도구
현실의 복잡한 객체들을 이해 가능한 구조로 만들기 때문.

---

# 3. 타입(Type)

● 타입은 곧 개념이다
	•	객체들의 공통점을 기반으로 묶어놓은 분류 기준.

● 데이터 타입의 본질
	•	메모리는 단지 비트 열 → 의미는 외부 프로그램(해석자)이 부여
	•	비트에 의미를 부여한 것 = 타입 시스템
→ 혼란 방지, 잘못된 사용 금지
	•	중요한 특징:
	•	타입은 데이터가 어떻게 사용되는가에 의해 정의됨
	•	내부 표현(비트)은 감춰진다 (추상화!)

---

# 4. 객체와 타입 — 행동이 핵심이다

● 타입을 결정하는 기준 = 행동(Responsibility)
	•	객체 내부에 어떤 데이터가 있는지보다
외부에 어떤 메시지를 처리할 수 있는지가 타입을 결정한다.
	•	같은 데이터라도 행동이 다르면 다른 타입
	•	다른 데이터라도 행동이 같으면 같은 타입

👉 이것이 다형성(polymorphism), **캡슐화(encapsulation)**의 근본 원리.

● 책임 주도 설계(RDD)
	1.	먼저 “어떤 책임을 가져야 하는가?”를 결정
	2.	그 책임을 수행하기 위한 데이터는 나중에 선택
	3.	내부 데이터는 인터페이스 뒤로 숨김 → 캡슐화

---

# 5. 타입의 계층(일반화/특수화)

● 트럼프 ↔ 트럼프 인간 예시
	•	트럼프 = 더 일반적인 타입(슈퍼타입)
	•	트럼프 인간 = 더 특수한 타입(서브타입)
	•	특수한 타입은:
	•	더 좁은 범위
	•	더 많은 행동을 가질 수 있음
	•	하지만 슈퍼타입이 할 수 있는 행동은 모두 가능해야 함 → 행위적 호환성

👉 상속의 본질은 데이터 재사용이 아니라 행동의 일반화/특수화 관계.

---

# 6. 정적 모델의 필요성
	•	실제 객체는 시간이 흐르며 상태가 계속 변하는데
인간은 이 동적 변화를 그대로 이해하기 어려움.
	•	그래서 객체의 행동을 기준으로 **시간과 무관한 정적 모델인 ‘타입’**을 만든다.
	•	프로그래밍 언어에서는 이 타입을 클래스로 구현할 수 있다.

---

# 📌 핵심 정리 한 줄 요약

타입은 객체의 “데이터”가 아니라 “행동(책임)”에 의해 정의된다.
추상화와 분류를 통해 복잡한 현실을 다루기 쉬운 구조로 바꾸는 것이 객체지향의 본질이다.
